# 手势和动画重构目标文档

## 目标

手势和动画的重构目标，是把已经形成的使用习惯整理清楚，让代码准确表达这些交互意图。

重构完成后，项目应该保持当前好用的部分，减少偶发 bug，降低后续修改手势时的误改风险。代码可以变多一点，但阅读路径要更清楚。后续改动应先看交互规则，再看具体实现，不再靠猜 `classList` 和历史补丁之间的关系。

## 核心体验

应用的手势和动画应该接近手机原生小工具的感觉：轻、快、可预期。

用户拖动时，界面要跟手。松手后，界面要自然完成打开、关闭或退回。点击打开时，面板要干净展开。任何时候都不应该出现闪开又闪关、残留位移、按钮点不动、阴影突然跳出来这类表现。

手势不应该抢用户操作。输入框、按钮、筛选器、打分按钮这些明确操作区域，应优先响应原本的点击或输入，不应被滑动误触发。

## 浮层关系

同一时间只让一个主要浮层占据注意力。

主要浮层包括：

- drawer：作业侧栏
- quickPanel：当前作业面板
- newAssignmentPanel：新建作业面板
- scoreSheet：打分面板
- confirmPanel：确认框
- settingsPanel：设置页
- rosterEditorPanel：名单编辑页
- toast：提示条

确认框优先级最高。确认框出现时，底下的主要滑动手势不应继续响应。

打分面板、设置页、名单编辑页属于明确任务，打开后应尽量稳定，不被其它入口抢走操作。

quickPanel 和 drawer 是高频入口，可以更灵活。一个浮层正在打开时，另一个不应插进来。一个浮层正在关闭时，允许用户马上打开另一个，这样连续操作会更快。

## quickPanel

quickPanel 是当前作业的快速操作面板，应该适合频繁打开和关闭。

期望行为：

- 学生列表顶部下拉，可以打开 quickPanel。
- 下拉时显示预览，距离够了松手打开，不够就退回。
- quickPanel 打开后，面板内部上滑可以关闭。
- quickPanel 打开后，面板外空白区域上滑也可以关闭。
- 操作记录视图里，不应因为轻微上滑误关面板。
- 主视图和操作记录视图切换时，quickPanel 高度保持稳定。
- 视图切换只切换内容，不改变外层面板高度。
- 内容比面板高时，在面板内部滚动。
- 视图切换可以有轻微淡入淡出，但不要有高度展开或收起感。
- 关闭 quickPanel 时，要清理下拉预览、视图切换状态、临时 transform 和拖动 class。

quickPanel 是手势里最复杂的部分。重构时应把它拆成四个明确动作：下拉预览、释放打开、面板内上滑关闭、面板外上滑关闭。

## drawer

drawer 是作业列表入口，不应该挡住正常滚动和点击。

期望行为：

- 屏幕左侧横滑可以打开 drawer。
- drawer 内部左滑可以关闭。
- drawer 外部空白区域左滑可以关闭。
- drawer 固定在底层，主页整体右移并显示左侧圆角；普通开合不移动 drawer 本身。
- drawer 关闭后不能挡住下面的学生列表。
- drawer 搜索框和科目筛选不触发横滑关闭。
- 切换作业时，drawer 先关闭，再切换内容。
- drawer 打开动画可以比 quickPanel 稍重，但不能拖慢操作。
- drawer 全屏展开相关动画继续保留，但要和普通打开、关闭状态分开管理。

drawer 的重点是不要误挡、不要误触发、不要在关闭后留下可点击遮挡。

## scoreSheet

scoreSheet 是打分输入面板，稳定比灵活重要。

期望行为：

- 点击学生卡片打开 scoreSheet。
- 打开后的短时间内防误触，避免刚点开就触发内部按钮。
- 下滑可以关闭 scoreSheet。
- 打开或关闭动画中可以按下暂停，并从当前高度反向接管。
- scoreSheet 打开时，其它主要入口不要抢手势。
- toast 不应穿透到底层影响 scoreSheet。
- 关闭 scoreSheet 时，分数输入状态、备注状态、当前学生状态要清理干净。
- 关闭后视口内不可见：默认 `visibility:hidden`（`components.css`），不依赖 transform 是否多停一帧；打开（`is-open`）与拖动/释放动画（`is-motion-dragging`）期间保持 `visible`。

scoreSheet 不需要复杂手势。可靠和少误触更重要。

## newAssignmentPanel

newAssignmentPanel 是新建作业面板，使用频率低于 quickPanel，但行为应和顶部面板保持一致。

期望行为：

- 点击新建按钮打开。
- 打开时关闭其它主要浮层。
- 上滑可以关闭。
- 打开或关闭动画中可以按下暂停，并从当前高度反向接管。
- 面板外空白区域上滑可以关闭。
- 打开后可以自动聚焦标题输入框。
- 关闭时清理输入焦点、阴影延后状态和 transform。

newAssignmentPanel 可以复用 top-sheet 的通用规则，但不应继承 quickPanel 的下拉打开逻辑。

## confirmPanel

confirmPanel 是最高优先级的临时确认层。

期望行为：

- 出现时阻止底下浮层的主要滑动手势。
- 返回键、Esc、Android 返回键优先关闭确认框。
- 关闭确认框后，底下原有浮层状态保持正常。
- 不把 confirmPanel 当成普通 drawer 或 sheet 处理。

确认框的规则要简单，优先保证不会误操作底下内容。

## toast

toast 是提示，不参与主要浮层关闭顺序。

期望行为：

- toast 可见时可以下滑关闭。
- toast 手势不应穿透到底层 sheet 或 panel。
- 自动消失和手势关闭不能互相打架。
- toast 不使用 `is-open` 判断，它只看自己的可见状态。

toast 应保持独立，不要混进主要浮层互斥逻辑。

## 动画表现

动画遵循四条规则：

- 拖动时跟手，不要延迟。
- 松手后有惯性，但不要飞太远。
- 动画结束后状态必须干净。
- 阴影只增强层级感，不应影响操作判断。

点击打开和拖动打开可以表现不同：

- 点击打开：面板完整展开，阴影稍后出现。
- 拖动打开：面板跟随手指，释放后自然落位，不额外叠加阴影延后。

拖动过程中可以临时关闭阴影，减少性能压力。静止打开后再显示阴影。

## 状态设计

重构后，每个主要浮层应有一套统一运动状态。

建议状态：

- `closed`：关闭
- `opening`：点击打开动画中
- `open`：已打开
- `dragging`：用户正在拖动
- `settling-open`：释放后正在落到打开位置
- `settling-close`：释放后正在落到关闭位置
- `closing`：点击或程序触发关闭动画中

现有 CSS class 可以继续保留，但它们应变成显示结果，不再作为分散的事实来源。

现有 class 的定位：

- `is-open`：视觉上处于打开态
- `is-dragging`：quickPanel 下拉预览态
- `is-motion-dragging`：拖动或释放动画期间的运动状态
- `is-expanding`：drawer 全屏展开中
- `no-anim`：瞬间重置，不播放动画
- `is-pointer-guarded`：scoreSheet 打开后的短暂防误触
- scoreSheet 关闭可见性：默认 `visibility:hidden`；`is-open` 或 `is-motion-dragging` 时为 `visible`（防 Android WebView 释放后一帧重画，见 CodeGraph「scoreSheet」）

这些 class 由统一状态模块写入和清理。四类面板的深度效果由共享 `#layerScrim` 表达，不再维护阴影延后 class。

## 手势开始规则

每个手势开始前，都要先判断三件事：

1. 当前浮层状态是否允许这个手势。
2. 当前触点区域是否允许这个手势。
3. 当前是否已有同一个浮层的释放动画在进行。

触点区域规则应集中表达，避免每个文件散写一串 `closest()`。

典型规则：

- 输入框、选择框、文本区域不触发面板手势。
- 普通按钮不触发下拉打开 quickPanel。
- 学生卡片可以参与列表顶部下拉。
- drawer 搜索和筛选不触发 drawer 横滑。
- quickPanel 操作记录视图不触发面板内上滑关闭。
- 面板外空白区允许触发关闭，但面板内部不重复绑定同一类关闭手势。

## 冲突处理

用户快速连续操作时，按这些规则处理：

- 正在打开的浮层，会阻止其它浮层打开。
- 正在关闭的浮层，不阻止其它浮层打开。
- drawer、quickPanel、newAssignmentPanel、scoreSheet 的开合动画未结束时，非控件区域按下可立即暂停；方向成立后从当前画面位置接管。
- 横滑和竖滑只能有一个方向拿到当前触点。
- `pointercancel` 发生时，界面退回干净状态。
- 释放动画被中断时，必须清理 transform、transition、willChange 和运动 class。
- 按下后未形成正确方向拖动、触点取消或应用切后台时，从暂停位置继续原动画目标。
- toast、确认框和设置/名单全屏切换不使用动画接管。
- 程序关闭浮层时，也要清理手势预览和共享遮罩状态。

这几条应成为重构后的核心规则。

## 不希望出现的表现

重构后应避免以下问题：

- 面板闪开又闪关。
- quickPanel 下拉一半后卡住。
- drawer 关闭后仍挡住学生列表。
- 松手后面板残留一段位移。
- 遮罩透明度与面板打开幅度不同步。
- 快速滑动后按钮点不动。
- Android 上轻滑触发错误关闭。
- 操作记录视图里误关 quickPanel。
- 确认框出现时底下浮层还能被滑动。
- 点击打开和拖动打开的状态互相污染。
- 关闭动画中留下 `is-motion-dragging` 或可见遮罩。

## 代码重构目标

第一阶段只做状态收拢，不改交互表现。

目标：

- 新增统一的浮层运动状态模块。
- 让拖动、释放动画、点击打开和共享遮罩都通过统一控制器协调。
- 保留现有 CSS class 输出。
- 保留现有动画时间、缓动曲线和视觉表现。
- 先不合并横向和纵向拖动引擎。
- 先不重写 CSS 动画。
- 先不改变 quickPanel、drawer、scoreSheet 的用户行为。

涉及文件以这些为主：

- `src/js/gestures/motion-registry.js`
- `src/js/gestures/drag-gesture.js`
- `src/js/gestures/interactive-layer-controller.js`
- 新增 `src/js/gestures/layer-motion-state.js`

第二阶段整理具体手势入口。

目标：

- 拆清 `panel-swipe.js` 中 quickPanel 的四类动作。
- 把手势开始判断改成命名函数。
- 把浮层互斥判断集中表达。
- 减少散落的 `classList.contains()` 和 `closest()`。
- 保持原有手动体验不变。

第三阶段再考虑减少重复。

目标：

- 对比垂直拖动和水平拖动的共同部分。
- 只在行为稳定后抽公共生命周期。
- 不为了减少行数提前抽象。
- 不把 quickPanel 的特殊规则强行塞进通用模块。

## 保留不动的内容

第一轮重构不动这些内容：

- `gesture-motion-engine.js` 的动画算法。
- CSS 的实际动画时间和缓动曲线。
- Android `touchmove preventDefault` 处理。
- `pointercancel` 处理。
- scoreSheet 的短暂防误触。
- 浮层关闭顺序。
- quickPanel 高度稳定的当前表现。
- toast 的独立可见状态判断。

这些内容现在承担实际体验，直接改动风险高。

## 验收方式

重构后需要手动验证，重点看真实操作感。

quickPanel：

- 学生列表顶部下拉，距离不够松手退回。
- 学生列表顶部下拉，距离够松手打开。
- quickPanel 内部上滑关闭。
- quickPanel 外部空白区域上滑关闭。
- 操作记录视图中轻微上滑不误关。
- 主视图和操作记录切换时，面板高度不变化。
- 快速下拉、松手、再滑动，不出现卡住或残留位移。

drawer：

- 左侧横滑打开 drawer。
- drawer 内部左滑关闭。
- drawer 外部空白区域左滑关闭。
- drawer 搜索框和筛选器可以正常输入、选择。
- drawer 关闭后，学生列表可以正常滚动和点击。
- 切换作业时 drawer 先关闭。

scoreSheet：

- 点击学生卡片打开。
- 打开后短时间内不误触内部按钮。
- 下滑关闭；关闭后视口底无闪现（Android WebView）。
- 打开 scoreSheet 后，其它入口不抢手势。
- toast 出现时不穿透影响 sheet。

浮层冲突：

- 确认框打开时，底下手势不响应。
- quickPanel 正在打开时，drawer 不插入打开。
- drawer 正在打开时，quickPanel 不插入打开。
- quickPanel 正在关闭时，可以马上打开 drawer。
- drawer 正在关闭时，可以马上下拉 quickPanel。
- 四类滑动面板释放动画中可以再次触摸接管；toast 释放中不重复接管。

异常操作：

- 快速来回滑动不残留 transform。
- 半路取消手势后界面恢复正常。
- Android WebView 上不闪关。
- 返回键、Esc、Android 返回键仍按原关闭顺序工作。

## 文档更新目标

重构完成后，`CodeGraph.md` 的手势部分应从“历史补丁说明”变成“交互规则说明”。

应保留：

- 每个浮层有哪些打开和关闭方式。
- 哪些浮层会阻止其它浮层打开。
- 哪些释放动画允许其它入口进入。
- 哪些 class 是视觉输出。
- 哪些状态由统一状态模块管理。

应减少：

- 对旧实现细节的长篇解释。
- 重复提醒某个 class 不能混用。
- 多处散写的特殊判断说明。

## 最终状态

重构完成后，手势和动画代码不一定很短，但应做到：

- 状态来源单一。
- 视觉 class 的职责清楚。
- 手势入口能直接看懂。
- 浮层互斥规则集中。
- quickPanel 的特殊行为被明确命名。
- 后续修改不需要在多个文件之间来回猜。
- 当前已经符合使用习惯的表现不被误改。
