# Plan Handoff Guide

用于把一次问题分析沉淀成可交给其他智能体执行的计划。目标不是展示推理过程，而是消除执行阶段的再决策成本。

## 设计思路

计划先回答事实，再回答动作。事实来自当前环境、代码、日志、用户路径和约束；动作只覆盖已被事实支持的最小改动。

一份可执行计划应满足以下条件：

- 已说明用户目标和可验收结果。
- 已定位当前状态，不能依赖记忆或猜测。
- 已明确根因或主要依据，避免只描述症状。
- 已选定一种实现路径，并说明为什么不改其它相邻逻辑。
- 已列出测试路径，包括用户原始复现步骤。
- 已写明假设、限制和已有改动，避免执行者覆盖无关工作。

弱模型执行计划时最容易出错的地方，是在模糊处自行补全。因此计划要把边界写清楚，尤其是“不改什么”和“为什么不改”。

## 计划生成流程

1. 收集环境事实。
   读取项目约束、入口文件、相关模块、当前工作区状态。先确认现有改动，避免把用户或其他工具的改动当作可回退内容。

2. 还原用户路径。
   把用户描述转成可重复的操作序列，逐步追踪事件、状态、渲染和副作用。每一步都要能落到具体函数、文件或条件。

3. 提炼根因。
   根因句要可验证。格式可采用“因为某文件某条件保留了某状态，所以某入口提前返回，导致某用户行为失效”。

4. 选择最小修复面。
   优先修改产生或清理错误状态的位置。保留原本合理的防护逻辑，不为了当前症状删除整个机制。

5. 写成执行计划。
   执行计划按行为分组，不按读取顺序堆文件。每条修改都要让执行者知道目标、边界和验收方式。

6. 做计划自查。
   检查是否存在未决选择、占位词、隐含前提、超过范围的重构、缺失的测试路径。

## 输出格式

正式计划使用 `<proposed_plan>` 包裹，便于客户端和其他智能体识别。计划内部保持 Markdown，结构短而完整。

```markdown
<proposed_plan>
# 标题

## Summary

用一小段说明目标、当前根因或主要依据、计划结果。这里要让执行者知道为什么这样改。

## Key Changes

- 按子系统或行为列出改动。
- 写清楚要改什么，也写清楚不改什么。
- 涉及公开 API、数据结构、配置、命令或持久化格式时，在这里明确说明。

## Test Plan

- 覆盖用户原始复现路径。
- 覆盖相邻路径，确认没有破坏原有防护。
- 列出可运行命令和手动验收点。

## Assumptions

- 写明执行依赖的前提。
- 写明已有未提交改动、环境限制、暂不自动化的原因。
- 写明默认取舍，避免执行者重新选择。
</proposed_plan>
```

## 写作规则

- 标题说明结果，不写过程口号。
- `Summary` 只放对执行有用的背景，避免复述全部调查过程。
- `Key Changes` 以行为为单位，少列文件清单；只有文件名能防止误改时才写文件名。
- `Test Plan` 同时包含原始 bug 路径和回归路径。
- `Assumptions` 只写会影响执行的前提，不写泛泛风险。
- 不使用 TBD、TODO、类似处理、后续决定等占位表达。
- 不把项目私有路径、一次性命令、临时日志方案沉淀为通用规则。

## 自查清单

交付前逐项检查：

- 执行者是否不需要再选择修复方向。
- 每个改动是否都能追溯到根因或验收目标。
- 是否说明了已有工作区改动，避免误回退。
- 是否保留了原有合理机制，只修正失效边界。
- 是否列出了至少一个用户路径验收项。
- 是否说明了不能自动验证的部分和原因。
- 计划是否能在不阅读对话上下文的情况下独立执行。

## 反例

以下写法不适合作为交接计划：

- “优化点击逻辑”，范围不清。
- “修复状态问题”，没有根因。
- “必要时调整相关文件”，让执行者重新决策。
- “跑一下测试”，没有具体命令或验收路径。
- “顺便清理代码”，混入未批准重构。

合格计划应让执行者只需要照计划改、照计划测、照计划汇报。
